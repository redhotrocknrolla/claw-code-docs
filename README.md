# Documentation Pipeline — Multi-Agent Skill System

Мультиагентный пайплайн для автоматической генерации технической документации по исходному коду. Работает через систему skill для GigaCode CLI.

> **Текущая ветка:** `weak-models-v3.0`
> Все скиллы оптимизированы для работы со слабыми языковыми моделями.

---

## Как это работает

Пайплайн состоит из специализированных агентов (skill), каждый из которых отвечает за свой этап. Orchestrator управляет последовательностью, выбирает нужный набор skill по языку и проверяет наличие артефактов на каждом шаге.

```
Git URL / локальный путь / файл
           │
           ▼
┌─────────────────────┐
│  git-repo-scanner   │  Шаг 0 — сканирование репо (опционально)
│  структура + история│  → repo-overview.md
└────────┬────────────┘
         ▼
┌─────────────────────┐
│    orchestrator     │  Определяет язык, имя проекта, набор skill
│    (IF/THEN дерево) │  Создаёт папки, ведёт журнал
└────────┬────────────┘
         ▼
┌─────────────────────┐
│  *-module-analyzer  │  Шаг 1 — извлечение фактов из кода
│  (карточки + чеклист│  → analyzer-output.md
└────────┬────────────┘
         ▼
┌─────────────────────┐
│    *-doc-writer     │  Шаг 2 — генерация черновика документации
│  (fill-in шаблоны) │  → doc-draft.md
└────────┬────────────┘
         ▼
┌─────────────────────┐
│  *-style-auditor    │  Шаг 3 — аудит фактов, стиля, структуры
│  (чеклист ДА/НЕТ)  │  → *-doc-advanced.md (финальный)
└────────┬────────────┘
         ▼
┌─────────────────────┐
│ confluence-publisher│  Шаг 4 — публикация в Confluence (опционально)
│  MCP или REST API   │  → publisher-session.md
└────────┬────────────┘
         ▼
┌─────────────────────┐
│       report        │  Шаг 5 — итоговый отчёт
│                     │  → *-report-YYYY-MM-DD.md
└─────────────────────┘
```

## Структура проекта (Variant B)

Каждый документируемый репозиторий получает изолированную директорию:

```
projects/<project-name>/
├── docs/
│   ├── work/           ← analyzer-output.md, doc-draft.md, repo-overview.md
│   ├── outputs/        ← <project-name>-doc-advanced.md
│   ├── dialogs/        ← orchestrator, analyzer, writer, auditor, scanner, publisher
│   └── notes/          ← advanced-note.md
└── reports/
    └── <project-name>-report-YYYY-MM-DD.md
```

## Оптимизация под слабые языковые модели (v3.0)

Версия `weak-models-v3.0` — полный рефакторинг скиллов под модели с ограниченными возможностями следования сложным инструкциям.

### Что было проблемой

| Проблема | Старый подход | Новый подход |
|----------|--------------|--------------|
| Ролевые переключения | `Analyst → Writer → Reviewer` в одном промте | Убраны. Заменены чеклистами |
| Многошаговые passes | 6 passes подряд без точек остановки | Атомарные шаги с СТОП-маркерами |
| Открытые инструкции | «Определи подтверждённые сценарии» | Fill-in карточки с конкретными полями |
| Нет примеров | Только правила | Примеры ✅ / ❌ для каждого ключевого места |
| Размытые требования | «Не скрывай ограничения» | Чеклист с ДА/НЕТ перед сохранением |
| Нет единого маркера | Разные фразы для неопределённости | Единый маркер `⚠️ Требует уточнения` |

### Ключевые принципы v3.0

**1. Атомарные шаги** — каждый шаг делает одно действие, ≤2 предложения инструкции.

**2. Fill-in карточки** — вместо «опиши метод» модель заполняет конкретные поля:
```
Сигнатура: [скопируй ТОЧНО из кода]
Возвращает: [тип / ⚠️ Требует уточнения]
Выбрасывает: [тип / нет / ⚠️]
```

**3. Примеры ✅ / ❌** — показываем что правильно и что нет прямо в скилле:
```
❌ Назначение: Обрабатывает пользовательские запросы
✅ Назначение: ⚠️ Требует уточнения — тело метода зависит от внешнего контекста
```

**4. СТОП-маркер** — явная точка «сохрани файл и передай управление». Модель не продолжает бесконечно.

**5. Чеклист ДА/НЕТ** — вместо ролевых переключений:
```
[ ] Все сущности описаны?
[ ] Нет запрещённых слов?
[ ] Все ⚠️ расставлены?
```

**6. Единый маркер неопределённости** — `⚠️ Требует уточнения` во всех скиллах.

**7. IF/THEN дерево в оркестраторе** — вместо «определи источник» — явные ветки:
```
→ Если Git URL → запусти git-repo-scanner
→ Если локальный файл → сразу к Шагу 1
→ Если неясно → задай один вопрос
```

## Скиллы-утилиты

| Скилл | Назначение |
|-------|-----------|
| `git-repo-scanner` | Сканирование структуры и истории Git-репозитория: языки, hotspots, changelog, breaking changes |

## Быстрый старт

1. Скопируй `.gigacode/skills/` в корень своего проекта или рабочей директории GigaCode.
2. Вызови `orchestrator`, передав URL или путь к репозиторию.
3. Orchestrator определит язык, создаст `projects/<name>/` и проведёт документирование шаг за шагом.
