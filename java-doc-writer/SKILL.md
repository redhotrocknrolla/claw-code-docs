# Java Doc Writer

## Правило №1 — главное

Пиши только то, что есть в `analyzer-output.md` или в коде.
Generics — копируй точно. Throws — перечисляй все из сигнатуры.
Аннотации — пиши как факт, не как описание поведения.
Если чего-то нет — пиши: `⚠️ Требует уточнения`

---

## Что ты получаешь на вход

- `docs/work/analyzer-output.md` — прочитай ПЕРВЫМ
- Исходный `.java` файл (если доступен)

Если `analyzer-output.md` не найден — сообщи и остановись.

---

## Шаг 1. Прочитай analyzer-output.md

Выпиши для себя:
- тип класса (class / interface / enum)
- количество методов
- список аннотаций класса
- зоны ⚠️ из анализа

---

## Шаг 2. Напиши заголовок класса

```markdown
### `<ИмяКласса>`

**Вид:** [class / interface / abstract class / enum]
**Пакет:** [com.example....]
**Наследование:** [extends X / implements Y, Z / нет]
**Аннотации класса:** [@Service / @Entity / нет]
**Назначение:** [1 предложение из кода / ⚠️ Требует уточнения]
```

---

## Шаг 3. Опиши каждый метод

Для каждого метода из `analyzer-output.md`:

```markdown
#### `<полная сигнатура метода>`

**Назначение:** [строго по коду / ⚠️ Требует уточнения]

**Параметры:**
| Имя | Тип | Описание |
|-----|-----|----------|
| name | String | ⚠️ Требует уточнения |

**Возвращает:** [тип + что именно / ⚠️]

**Исключения:**
- `CheckedException` — [когда / ⚠️]
- `RuntimeException` — [когда / ⚠️]

**Аннотации:** `@Override`, `@Transactional`
*(аннотации записаны как факт без описания runtime-поведения)*

**Пример:**
```java
// пример вызова если очевиден из кода
// если нет: "Пример недоступен: поведение зависит от контекста ⚠️"
```
```

**Запрещённые слова** — проверь каждый метод:
`удобно`, `просто`, `легко`, `очевидно`, `гибкий`, `мощный`, `полезный`, `продвинутый`, `по сути`, `фактически`, `магия`, `штука`

---

## Шаг 4. Поля и константы

Скопируй таблицу из `analyzer-output.md` раздел 3.
Добавь колонку «Описание» — только если понятно из кода.

```markdown
#### Поля и константы

| Имя | Тип | Модификаторы | Аннотации | Описание |
|-----|-----|-------------|-----------|----------|
```

---

## Шаг 5. Сценарии работы

Берёшь сценарии из `analyzer-output.md`.
Если их нет — пиши: `Сценарии не определяются без внешнего контекста ⚠️`

Если есть — оформи:
```markdown
### Сценарий: [название]
1. `метод(параметр)` вызывается с [условие]
2. Если [условие] → выбрасывается `ExceptionType`
3. Иначе → возвращается [тип]

Источник: `ИмяМетода`
```

---

## Шаг 6. Ограничения

Пиши только то, что ЯВНО видно в коде:
- `throws` в сигнатурах
- `if (x == null) throw new ...`
- `@NotNull`, `@NonNull` аннотации на параметрах

❌ Не пиши: «Класс не является потокобезопасным» (если этого нет в коде)
✅ Пиши: «Метод `save()` выбрасывает `DataIntegrityException` при нарушении уникальности»

---

## Шаг 7. Проверка перед сохранением

```
[ ] Все методы из analyzer-output.md описаны?
[ ] Generics скопированы точно (не упрощены)?
[ ] Все checked exceptions из throws перечислены?
[ ] Аннотации записаны без runtime-объяснений?
[ ] Нет запрещённых слов?
[ ] Все ⚠️ перенесены из анализа?
[ ] Нет ни одного утверждения без источника в коде?
```

Если хотя бы один — НЕТ: исправь.
