# Архитектурное пояснение advanced-пайплайна документирования

## Общая архитектура

Advanced-пайплайн документирования модулей Rust реализует многокомпонентную систему с разделением ответственности:

```
┌─────────────────────────────────────────────────────────────────────┐
│                        Orchestrator (GigaCode)                     │
│  Координирует весь процесс, управляет состоянием и валидацией      │
└─────────────────┬───────────────────────────────────────────────────┘
                  │
        ┌─────────┼─────────┐
        │         │         │
        ▼         ▼         ▼
┌─────────────┐ ┌─────────────┐ ┌──────────────┐
│   Analyzer  │ │   Writer    │ │   Auditor    │
│  Rust-Mod   │ │  Rust-Doc   │ │ Rust-Style   │
└─────────────┘ └─────────────┘ └──────────────┘
        │         │         │
        └─────────┼─────────┘
                  ▼
        ┌─────────────────┐
        │   Outputs       │
        │   (Docs)        │
        └─────────────────┘
```

## Компоненты пайплайна

### 1. Orchestrator (GigaCode)

**Роль:** Главный координатор пайплайна

**Задачи:**
- Проверка наличия всех необходимых skills
- Создание и управление структурой каталогов для артефактов
- Запуск этапов в правильном порядке
- Валидация промежуточных результатов
- Формирование финального журнала

**Выходные файлы:**
- `docs/dialogs/orchestrator-session.md` — общий журнал

### 2. Analyzer (rust-module-analyzer)

**Роль:** Анализ исходного кода модуля

**Задачи:**
- Поиск всех публичных сущностей (struct, enum, fn, impl)
- Извлечение сигнатур и полей
- Выявление зависимостей и связей
- Определение непубличных, но значимых элементов
- Обнаружение мест повышенного риска интерпретации

**Выходные файлы:**
- `docs/work/analyzer-output.md` — подробный анализ модуля
- `docs/dialogs/analyzer-session.md` — журнал сессии

### 3. Writer (rust-doc-writer)

**Роль:** Генерация черновика документации

**Задачи:**
- Формирование структуры документа по требованиям
- Оформление каждой сущности в официальном стиле
- Создание описаний сценариев работы
- Обозначение ограничений
- Добавление примечаний по использованию

**Выходные файлы:**
- `docs/work/doc-draft.md` — черновик документации
- `docs/dialogs/writer-session.md` — журнал сессии

### 4. Auditor (rust-style-auditor)

**Роль:** Аудит финальной документации

**Задачи:**
- Проверка соответствия критериям качества
- Проверка соблюдения структуры документа
- Проверка языка (русский, официальный тон)
- Убедиться в отсутствии разговорной лексики
- Проверить пометки "Требует уточнения"

**Выходные файлы:**
- `docs/outputs/tools-lib-doc-advanced.md` — финальная документация
- `docs/dialogs/auditor-session.md` — журнал сессии

## Критерии качества

1. **Только модуль rust/crates/tools/src/lib.rs** — документация только для указанного модуля
2. **Только подтвержденные кодом утверждения** — нет предположений без кодовой базы
3. **Русский язык** — все тексты на русском языке
4. **Официальный технический тон** — без разговорных выражений и рекламной лексики
5. **Единая структура документации** — все документы следуют одному шаблону
6. **Отсутствие разговорной и рекламной лексики** — строго технический стиль
7. **Неочевидные места помечены** — все зависимости от других модулей отмечены

## Ключевые решения архитектуры

### Разделение ответственности
Каждый компонент отвечает за свою часть работы, что позволяет:
- Тестировать компоненты независимо
- Заменять компоненты без переписывания всей системы
- Повторно использовать компоненты для других модулей

### Валидация на каждом этапе
После каждого этапа проверяется, что выходной файл создан и имеет правильную структуру. Это позволяет выявлять ошибки на ранних стадиях.

### Формальный подход к документированию
Использование структурированных шаблонов и таблиц обеспечивает единообразие документации и облегчает поиск информации.

## Использованные инструменты

- **GigaCode (skill-router)** — главный координатор
- **rust-module-analyzer** — анализ исходного кода
- **rust-doc-writer** — генерация документации
- **rust-style-auditor** — аудит качества

## Путь выполнения

1. Проверка skills
2. Создание структуры каталогов
3. Загрузка исходного кода
4. Анализ (analyzer → analyzer-output.md)
5. Генерация черновика (writer → doc-draft.md)
6. Аудит и финализация (auditor → tools-lib-doc-advanced.md)
7. Создание журналов сессий
8. Финальная проверка всех артефактов

## Заметки по расширению

Архитектура позволяет легко расширять пайплайн:
- Добавление новых этапов анализа
- Изменение формата выходной документации
- Интеграция с другими системами документирования
- Добавление новых критериев качества
