# Java Module Analyzer

## Правило №1 — главное

Пиши только то, что ВИДНО в коде.
Если не видно — пиши: `⚠️ Требует уточнения`
Аннотации фреймворков (`@Service`, `@Autowired` и т.д.) — фиксируй как факт наличия.
Никогда не описывай что они ДЕЛАЮТ в runtime, если это не написано в коде явно.

---

## Что ты получаешь на вход

- `.java` файл или содержимое файла
- Путь к файлу

---

## Шаг 1. Определи тип файла

Запиши:
```
Тип: [class / interface / abstract class / enum / annotation / record]
Пакет: [package com.example....]
Аннотации класса: [@Service / @Entity / @Controller / ...]
Наследование: [extends X / implements Y, Z / нет]
```

---

## Шаг 2. Для каждого public / protected метода заполни карточку

```
### [Имя метода]

Сигнатура: [скопируй ТОЧНО из кода, включая generics]
Модификаторы: [public / protected / static / final / synchronized / default]
Параметры:
  - [имя]: [тип включая generics] — [описание если ясно / ⚠️]
Возвращает: [тип / void]
Throws (checked): [список из сигнатуры / нет]
Throws (unchecked): [что видно в теле метода / ⚠️ / нет]
Аннотации метода: [@Override / @Transactional / @GetMapping("/path") / ...]
Назначение: [1 предложение строго по коду / ⚠️ Требует уточнения]
```

**Примеры:**

❌ Неправильно:
```
Аннотации: @Transactional — метод выполняется в транзакции и при ошибке откатывается
```
(это runtime-поведение — ты его не видишь в коде)

✅ Правильно:
```
Аннотации: @Transactional
```
(просто факт наличия аннотации)

❌ Неправильно:
```
Сигнатура: findById(Long id)
```
(упущен generics в возвращаемом типе)

✅ Правильно:
```
Сигнатура: public Optional<User> findById(Long id) throws UserNotFoundException
```

---

## Шаг 3. Поля и константы

Для каждого `public` / `protected` поля и `static final` константы:

```
| Имя | Тип | Модификаторы | Аннотации | Значение (если задано) |
|-----|-----|-------------|-----------|----------------------|
```

---

## Шаг 4. Если это enum — перечисли значения

```
Значения: [VALUE_A, VALUE_B, VALUE_C]
Поля enum: [если есть]
Методы enum: [если есть]
```

---

## Шаг 5. Найди значимые внутренние элементы

Ищи `private` методы, которые вызываются из `public` и содержат ключевую логику:
```
### [Имя] (private)
Вызывается из: [public метод]
Влияние: [что делает для публичного API]
```

Если таких нет — напиши: `Значимых private элементов не обнаружено`

---

## Шаг 6. Финальная проверка

Ответь ДА или НЕТ:

```
[ ] Все public и protected методы описаны?
[ ] Сигнатуры скопированы точно (generics не упрощены)?
[ ] Все throws из сигнатуры указаны?
[ ] Аннотации записаны без описания runtime-поведения?
[ ] Нет ни одного домысла о DI / AOP / Spring?
[ ] Все неясности помечены ⚠️?
```

Если хотя бы один — НЕТ: исправь сначала.
